# Доповідь: Порушення принципу LSP

### 1. Птах, що не літає (Звуження поведінки)
* **Опис:** Базовий клас `Bird` має метод `Fly()`. Підклас `Ostrich` (Страус) викидає помилку в цьому методі.
* **Чому порушує:** Клієнт очікує, що всі птахи літають. Якщо замінити `Bird` на `Ostrich`, програма впаде.
* **Рішення:** Створити окремий інтерфейс `IFlyable` лише для тих птахів, що літають.

### 2. Тимчасовий працівник (Посилення передмов)
* **Опис:** Базовий клас `Employee` розраховує бонус усім. Підклас `Contractor` вимагає додаткової перевірки завершення контракту перед розрахунком.
* **Чому порушує:** Підклас висуває жорсткіші вимоги до вхідних даних/стану, ніж базовий клас.
* **Рішення:** Винести розрахунок бонусу в окремий інтерфейс `IBonusEligible`.

### 3. ReadOnly файл (Зміна стану)
* **Опис:** Клас `File` має метод `Write()`. Підклас `ReadOnlyFile` ігнорує запис.
* **Чому порушує:** Код, що працює з файлом, очікує збереження даних. Підміна на ReadOnly-версію призведе до втрати даних без попередження.
* **Рішення:** Розділити інтерфейси на `IReadable` та `IWritable`.
