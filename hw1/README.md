# Доповідь: SRP та анти-патерн God Object

### 1. Характеристики "God Object"
**God Object (Об'єкт-Бог)** — це об'єкт, який концентрує в собі занадто багато функцій системи.
* **Порушення інкапсуляції**: він втручається в роботу майже всіх інших класів.
* **Складність підтримки**: зміна однієї маленької функції може зламати всю програму.
* **Неможливість тестування**: через величезну кількість залежностей неможливо написати чисті Unit-тести.

### 2. Приклад порушення SRP (клас BadUserAccount)
Клас `BadUserAccount` є типовим прикладом порушення SRP, оскільки він має наступні обов'язки:
* **Керування даними**: збереження полів `Username` та `Email`.
* **Безпека**: метод `HashPassword` для шифрування.
* **Валідація**: перевірка формату пошти в `ValidateEmail`.
* **Інфраструктура**: відправка листів (`SendWelcomeEmail`) та SQL-запити до БД (`SaveToDatabase`).

### 3. Рефакторинг
Для дотримання SRP ми розділили обов'язки між різними компонентами:
* `IOrderValidator`: відповідає тільки за перевірку даних.
* `IOrderRepository`: відповідає тільки за збереження (Data Access).
* `IEmailService`: відповідає тільки за комунікацію з клієнтом.
* `OrderService`: виступає координатором, не знаючи деталей реалізації кожного процесу.
